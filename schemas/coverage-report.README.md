# Coverage Report Schema

## Overview

The `coverage-report.schema.json` defines the structure for requirement coverage analysis reports generated by the `req-coverage` tool. This schema validates JSON output that tracks how test cases cover requirements, including coverage statistics and test execution results.

## Schema Location

```
schemas/coverage-report.schema.json
```

## Purpose

- Validate requirement coverage reports
- Track coverage statistics (full, partial, uncovered requirements)
- Map test cases to requirements
- Record test execution status (pass, fail, not executed)

## Related Tool

**req-coverage** - Analyzes test cases and verification results to generate coverage reports

### Tool Commands

```bash
# Generate coverage report
req-coverage verify \
  --test-cases-folder testcases/ \
  --test-results-folder logs/ \
  --output coverage-report.json

# Generate HTML report from coverage JSON
req-coverage print \
  --format html \
  --input coverage-report.json \
  --output reports/
```

## Data Structure

### Root Object: CoverageReport

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `generated_at` | string (ISO 8601) | Yes | Timestamp when the report was generated |
| `summary` | object | Yes | Summary statistics of requirement coverage |
| `requirements` | array | Yes | List of requirement coverage items |

### Summary Object

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `total_requirements` | integer (≥0) | Yes | Total number of requirements analyzed |
| `fully_covered` | integer (≥0) | Yes | Number of requirements with full coverage |
| `partially_covered` | integer (≥0) | Yes | Number of requirements with partial coverage |
| `uncovered` | integer (≥0) | Yes | Number of requirements with no coverage |

### RequirementCoverageItem

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `requirement_id` | string | Yes | Unique identifier for the requirement |
| `coverage_type` | enum | Yes | Type of coverage: `"full"` or `"partial"` |
| `test_cases` | array | Yes | List of test cases covering this requirement |
| `status` | enum | Yes | Coverage status (see below) |

### CoverageStatus Enum

| Value | Description |
|-------|-------------|
| `covered_pass` | Fully covered, all tests passed |
| `covered_fail` | Fully covered, some tests failed |
| `partial_covered_pass` | Partially covered, all tests passed |
| `partial_covered_fail` | Partially covered, some tests failed |
| `uncovered` | No coverage (no test cases) |

### TestCaseResult

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `test_case_id` | string | Yes | Unique identifier for the test case |
| `status` | enum | Yes | Execution status: `"pass"`, `"fail"`, or `"notexecuted"` |
| `covers` | string | No | Additional coverage information |
| `description` | string | No | Description of the test case |

## Validation

### Using validate-json Tool

```bash
# Validate a coverage report
validate-json coverage-report.json coverage-report.schema.json

# Validate with verbose output
validate-json coverage-report.json coverage-report.schema.json --verbose
```

### Using Node.js (ajv)

```javascript
const Ajv = require('ajv');
const ajv = new Ajv();
const schema = require('./schemas/coverage-report.schema.json');
const report = require('./coverage-report.json');

const valid = ajv.validate(schema, report);
if (!valid) {
  console.error('Validation errors:', ajv.errors);
}
```

### Using Python (jsonschema)

```python
import json
import jsonschema

# Load schema and report
with open('schemas/coverage-report.schema.json') as f:
    schema = json.load(f)

with open('coverage-report.json') as f:
    report = json.load(f)

# Validate
try:
    jsonschema.validate(instance=report, schema=schema)
    print("✓ Validation successful!")
except jsonschema.ValidationError as e:
    print(f"✗ Validation failed: {e.message}")
```

## Coverage Calculation Logic

The `req-coverage` tool calculates summary statistics as follows:

1. **summary.total_requirements:** Count of all unique requirement IDs
2. **summary.fully_covered:** Requirements with `coverage_type: "full"` and at least one test case
3. **summary.partially_covered:** Requirements with `coverage_type: "partial"` and at least one test case
4. **summary.uncovered:** Requirements with empty `test_cases` array

The `status` field for each requirement combines coverage type and test results:
- `covered_pass`: Full coverage, all tests passed
- `covered_fail`: Full coverage, at least one test failed
- `partial_covered_pass`: Partial coverage, all tests passed
- `partial_covered_fail`: Partial coverage, at least one test failed
- `uncovered`: No test cases (regardless of coverage type)

## Common Validation Errors

### Invalid Coverage Type

```json
{
  "coverage_type": "complete"  // ✗ Invalid
}
```

**Fix:** Use `"full"` or `"partial"`

### Invalid Status

```json
{
  "status": "passed"  // ✗ Invalid
}
```

**Fix:** Use one of: `"covered_pass"`, `"covered_fail"`, `"partial_covered_pass"`, `"partial_covered_fail"`, `"uncovered"`

### Invalid Test Status

```json
{
  "test_cases": [
    {
      "test_case_id": "TC-001",
      "status": "success"  // ✗ Invalid
    }
  ]
}
```

**Fix:** Use `"pass"`, `"fail"`, or `"notexecuted"`

### Missing Required Fields

```json
{
  "generated_at": "2024-01-20T15:30:45Z",
  "requirements": []
  // ✗ Missing: summary object
}
```

**Fix:** Include all required fields (generated_at, summary, requirements)

## Integration with TCMS

While this schema is standalone and not part of the versioned TCMS envelope system, it integrates with TCMS components:

- **Input:** Test case YAML files (validated by `test-case.schema.json`)
- **Input:** Verification container YAML files (from `verifier` tool)
- **Output:** Coverage report JSON (validated by this schema)
- **Output:** HTML reports (generated from coverage JSON)
