# Test Case Results Container Schema

This directory contains the JSON schema for validating test case result container outputs generated by the verifier tool.

## Canonical Schema Paths

**Working Schema (used by verifier)**: `data/testcase_results_container/schema.json`
**Versioned Schema (recommended for new development)**: `schemas/tcms/test-results-container.schema.v1.json`
**Sample Data (canonical location)**: `schemas/tcms/samples/testcase_results_container_sample.yml`

> **Note**: The schema in this directory (`data/testcase_results_container/schema.json`) is the working schema used by the verifier tool for backward compatibility. For new development, consider using the versioned schema with envelope support: `schemas/tcms/test-results-container.schema.v1.json`.

## Overview

The schema validates the structure and content of YAML/JSON reports produced by the `verifier` binary when using the container output format. This ensures that generated reports conform to a standardized structure for downstream processing, documentation generation, and integration with other systems.

## Schema File

**File**: `schema.json` (working schema, backward compatible)

**Purpose**: Defines the structure and validation rules for test case result container reports

## Schema Structure

The schema validates container reports with the following top-level structure:

```yaml
title: "Test Execution Results"              # Required: Report title
project: "Test Case Manager"                  # Required: Project name
test_date: "2024-03-16T12:00:00Z"            # Required: ISO 8601 timestamp
test_results:                                 # Required: Array of test case results
  - test_case_id: "TC001"
    description: "Sample test case"
    sequences: [...]                          # Test sequences with step results
    total_steps: 10
    passed_steps: 8
    failed_steps: 2
    not_executed_steps: 0
    overall_pass: false
metadata:                                     # Required: Execution metadata
  environment: "Production"                   # Optional
  platform: "Linux x86_64"                    # Optional
  executor: "Jenkins v3.2"                    # Optional
  execution_duration: 45.2                    # Required: seconds
  total_test_cases: 5                         # Required
  passed_test_cases: 3                        # Required
  failed_test_cases: 2                        # Required
```

## Validation

### Automatic Validation

The verifier automatically validates all generated output against this schema when:

1. **YAML Output Format**: `verifier -f logs/ --format yaml --output report.yaml`
2. **JSON Output Format**: `verifier -f logs/ --format json --output report.json`

### Validation Process

1. **Schema Loading**: Loads `data/testcase_results_container/schema.json`
2. **Schema Compilation**: Compiles the JSON schema using the `jsonschema` library
3. **Format Conversion**: For YAML output, converts to JSON for validation
4. **Schema Validation**: Validates the output structure and data types
5. **Error Reporting**: Reports detailed validation errors if the output doesn't conform

## Schema Features

### Required Fields

- **Container Level**: `title`, `project`, `test_date`, `test_results`, `metadata`
- **Test Result Level**: `test_case_id`, `description`, `sequences`, `total_steps`, `passed_steps`, `failed_steps`, `not_executed_steps`, `overall_pass`
- **Sequence Level**: `sequence_id`, `name`, `step_results`, `all_steps_passed`
- **Step Result Level**: `status`, `step`, `description`
- **Metadata Level**: `execution_duration`, `total_test_cases`, `passed_test_cases`, `failed_test_cases`

### Optional Fields

- **Container Level**: `metadata.environment`, `metadata.platform`, `metadata.executor`
- **Test Result Level**: `requirement`, `item`, `tc`
- **Sequence Level**: `requirement`, `item`, `tc`
- **Step Result Level**: `requirement`, `item`, `tc`, `expected`, `actual_result`, `actual_output`, `reason`

### Conditional Requirements

- **Failed Steps**: When `status` is `"fail"`, the following fields are required:
  - `expected` (object with `result` and `output`)
  - `actual_result`
  - `actual_output`
  - `reason`

### Data Types and Constraints

- **Strings**: 
  - `title`: 1-200 characters
  - `project`: 1-200 characters
  - `environment`: 1-100 characters (optional)
  - `platform`: 1-150 characters (optional)
  - `executor`: 1-150 characters (optional)

- **Integers**:
  - `sequence_id`: minimum 1
  - `step`: minimum 1
  - All count fields (`total_steps`, `passed_steps`, etc.): minimum 0

- **Numbers**:
  - `execution_duration`: minimum 0

- **Enumerations**:
  - `status`: must be one of `"pass"`, `"fail"`, or `"not_executed"`

- **Date/Time**:
  - `test_date`: ISO 8601 format (e.g., `"2024-03-16T12:00:00Z"`)

## Usage Examples

### Generate and Validate YAML Output

```bash
# Generate YAML report with automatic validation (uses default schema)
verifier -f logs/ --format yaml --output report.yaml

# With custom metadata
verifier -f logs/ --format yaml --output report.yaml \
  --title "Nightly Test Run" \
  --project "Production Suite" \
  --environment "Production" \
  --platform "Linux x86_64" \
  --executor "Jenkins v3.2"

# Using a custom schema file
verifier -f logs/ --format yaml --output report.yaml \
  --schema /path/to/custom/schema.json
```

### Generate and Validate JSON Output

```bash
# Generate JSON report with automatic validation (uses default schema)
verifier -f logs/ --format json --output report.json

# Using config file
verifier -f logs/ --format json --output report.json \
  --config verifier-config.yaml

# Using a custom schema file
verifier -f logs/ --format json --output report.json \
  --schema /path/to/custom/schema.json
```

### Troubleshooting Validation Errors

If validation fails:

1. **Check Required Fields**: Ensure all required fields are present
2. **Verify Data Types**: Confirm integers, strings, and numbers match schema requirements
3. **Review Constraints**: Check string length limits and numeric minimums
4. **Validate Enums**: Ensure status values are exactly `"pass"`, `"fail"`, or `"not_executed"`
5. **Check Date Format**: Verify `test_date` is in ISO 8601 format

## Schema Maintenance

### Updating the Schema

When modifying the schema:

1. Update `schema.json` with new fields or constraints
2. Update this README documentation
3. Test schema changes with sample outputs
4. Verify backward compatibility with existing reports

Currently using JSON Schema Draft 07. Future versions should maintain backward compatibility or provide migration tools.
