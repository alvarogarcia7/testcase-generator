# Test Plan Documentation Generator Compatibility

## Overview

This document describes the compatibility verification system for container YAML files that are consumed by the `test-plan-doc-gen` tool. The system ensures that verification results generated by the verifier can be properly formatted and consumed by the documentation generation pipeline.

## Purpose

The compatibility verification system serves to:

1. **Validate Container Format**: Ensure container YAML files match the expected schema for `test-plan-doc-gen`
2. **Detect Schema Mismatches**: Identify any incompatibilities between verifier output and documentation generator input
3. **Generate Compatibility Reports**: Produce detailed reports documenting validation results and issues
4. **Support CI/CD Integration**: Enable automated validation in continuous integration pipelines

## Components

### 1. Container YAML Schema

**Location**: `data/testcase_results_container/schema.json`

The JSON schema defines the structure expected by `test-plan-doc-gen`:

- **Required Fields**: `test_results` array (must not be empty)
- **Recommended Fields**: `title`, `project`, `test_date`, `metadata`
- **Test Result Structure**: Each test result contains sequences with step results
- **Step Result Variants**: Pass, Fail, NotExecuted enum variants

See `data/testcase_results_container/README.md` for detailed schema documentation.

### 2. Compatibility Checker Binary

**Binary**: `test-plan-documentation-generator-compat`

**Source**: `src/bin/test-plan-documentation-generator-compat.rs`

A command-line tool that validates container YAML files and generates compatibility reports.

#### Commands

##### Validate a Single File

```bash
cargo run --bin test-plan-documentation-generator-compat -- validate container.yaml
```

Output:
```
✓ Valid: container.yaml
  Test Results: 5
  Total Steps: 42
  Has Title: true
  Has Project: true
  Has Test Date: true
  Has Metadata: true
```

##### Batch Validation

```bash
cargo run --bin test-plan-documentation-generator-compat -- batch reports/ --recursive
```

Validates all container YAML files in a directory (optionally recursive).

##### Test Verifier Scenarios

```bash
cargo run --bin test-plan-documentation-generator-compat -- test-verifier-scenarios
```

Tests compatibility with verifier-generated output from `testcases/verifier_scenarios`.

##### Generate Compatibility Report

```bash
cargo run --bin test-plan-documentation-generator-compat -- report reports/ \
  --output compatibility_report.md \
  --format markdown
```

Generates a comprehensive compatibility report in Markdown or text format.

#### Options

- `--verbose` / `-v`: Show detailed validation information
- `--json`: Output results as JSON
- `--recursive` / `-r`: Recursive directory search (for batch command)
- `--continue-on-error` / `-c`: Continue validation on errors (for batch command)
- `--generate` / `-g`: Generate missing container files (for test-verifier-scenarios)

### 3. Compatibility Test Script

**Script**: `scripts/test_container_yaml_compatibility.sh`

**Makefile Target**: `make test-container-compat`

A comprehensive test script that:

1. Builds required binaries (verifier, compatibility checker)
2. Runs verifier on test scenarios to generate verification results
3. Converts verification JSON to container YAML format
4. Validates all container YAML files
5. Generates detailed compatibility reports

#### Usage

```bash
# Using make
make test-container-compat

# Direct invocation
./scripts/test_container_yaml_compatibility.sh
```

#### Output

The script generates:
- Individual verification JSON files for each test case
- Individual result YAML files for each test case
- Compatibility validation results
- Markdown and text format compatibility reports
- Summary statistics

Output directory: `reports/container_yaml_compatibility/`

## Validation Rules

### Errors (Fail Validation)

These issues will cause validation to fail:

1. **Missing test_results**: The `test_results` array is missing or empty
2. **Empty test_case_id**: A test result has an empty `test_case_id`
3. **Empty sequences**: A test result has an empty `sequences` array
4. **Invalid step format**: A step result is not a mapping object
5. **Invalid variant structure**: Step result has no variant key or multiple variant keys

### Warnings (Informational)

These issues are reported but don't cause validation failure:

1. **Missing recommended fields**: `title`, `project`, `test_date`, or `metadata` are missing
2. **Empty sequence names**: A sequence has an empty name
3. **Empty step_results**: A sequence has an empty `step_results` array
4. **Missing Fail fields**: A Fail variant is missing recommended fields (`expected`, `actual_result`, `reason`)

### Compatibility Issues (Informational)

These issues indicate potential compatibility problems:

1. **Step count mismatch**: `passed_steps + failed_steps + not_executed_steps ≠ total_steps`
2. **Unknown variant**: Step result uses a variant other than Pass/Fail/NotExecuted
3. **Multiple variants**: Step result has multiple variant keys

## Container YAML Format

### Minimal Valid Container

```yaml
test_results:
  - test_case_id: 'TC_001'
    sequences:
      - sequence_id: 1
        name: "Test Sequence"
        step_results:
          - Pass:
              step: 1
              description: "Step description"
        all_steps_passed: true
    total_steps: 1
    passed_steps: 1
    failed_steps: 0
    not_executed_steps: 0
    overall_pass: true
```

### Complete Container with Metadata

```yaml
title: 'Test Execution Results'
project: 'Test Case Manager'
test_date: '2024-03-15T14:30:00Z'
test_results:
  - test_case_id: 'TC_001'
    description: 'Example test case'
    requirement: 'REQ_100'
    item: 1
    tc: 1
    sequences:
      - sequence_id: 1
        name: "Test Sequence #01"
        step_results:
          - Pass:
              step: 1
              description: "Execute command"
          - Fail:
              step: 2
              description: "Verify output"
              expected:
                success: true
                result: "0"
                output: "Success"
              actual_result: "1"
              actual_output: "Error"
              reason: "Exit code mismatch: expected '0', got '1'"
        all_steps_passed: false
    total_steps: 2
    passed_steps: 1
    failed_steps: 1
    not_executed_steps: 0
    overall_pass: false
metadata:
  environment: 'Test Environment'
  platform: 'Test Case Manager v1.0'
  executor: 'Automated Test Framework'
  execution_duration: 123.45
  total_test_cases: 1
  passed_test_cases: 0
  failed_test_cases: 1
```

## Integration with Verifier

### Workflow

The compatibility system integrates with the verifier workflow:

```
1. Test Case YAML
       ↓
2. Test Execution (generates execution log JSON)
       ↓
3. Verifier (generates verification result JSON)
       ↓
4. convert_verification_to_result_yaml.py (generates result YAML)
       ↓
5. Container Aggregation (combines into container YAML)
       ↓
6. test-plan-documentation-generator-compat (validates container)
       ↓
7. test-plan-doc-gen (generates documentation)
```

### Converting Verifier Output

Use the conversion script to transform verifier JSON output to container YAML:

```bash
# Convert individual verification result
python3 scripts/convert_verification_to_result_yaml.py \
  verification.json \
  -o results/ \
  -v

# This generates:
# - results/{test_case_id}_result.yaml (with 'type: result' field)
```

### Aggregating Result Files

To create a container YAML from multiple result files:

```bash
# Create container header
cat > results_container.yaml << 'EOF'
title: 'Test Results'
project: 'Test Case Manager'
test_date: '2024-03-15T14:30:00Z'
test_results:
EOF

# Append result files (remove 'type: result' line)
for file in results/*_result.yaml; do
    sed '/^type: result/d' "$file" | sed 's/^/  /' >> results_container.yaml
done

# Add metadata
cat >> results_container.yaml << 'EOF'
metadata:
  environment: 'Test Environment'
  total_test_cases: 5
EOF
```

## Common Issues and Solutions

### Issue: Empty test_results Array

**Symptom**: Validation error "Missing or empty 'test_results' array"

**Solution**: Ensure at least one test result is included in the container:

```yaml
test_results:
  - test_case_id: 'TC_001'
    sequences: [...]
    total_steps: 1
    # ... other required fields
```

### Issue: Step Count Mismatch

**Symptom**: Compatibility issue about step count mismatch

**Solution**: Verify that step counts are calculated correctly:

```python
total_steps = passed_steps + failed_steps + not_executed_steps
```

### Issue: Missing Fail Variant Fields

**Symptom**: Warnings about missing fields in Fail variant

**Solution**: Include recommended fields for comprehensive failure reporting:

```yaml
Fail:
  step: 2
  description: "Verify output"
  expected:
    success: true
    result: "0"
    output: "Success"
  actual_result: "1"
  actual_output: "Error"
  reason: "Exit code mismatch"
```

### Issue: Unknown Step Variant

**Symptom**: Compatibility issue about unknown variant

**Solution**: Use only the three supported variants: Pass, Fail, NotExecuted

```yaml
# Valid variants:
- Pass: { step: 1, description: "..." }
- Fail: { step: 2, description: "...", ... }
- NotExecuted: { step: 3, description: "..." }

# Invalid:
- Skipped: { ... }  # Use NotExecuted instead
- Warning: { ... }  # Use Pass or Fail with notes
```
