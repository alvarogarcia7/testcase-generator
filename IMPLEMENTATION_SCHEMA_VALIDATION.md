# Schema Validation Implementation

This document describes the implementation of JSON schema validation for test case result container outputs generated by the verifier.

## Overview

The implementation adds automatic validation of YAML/JSON output from the `verifier` binary against a JSON schema, ensuring that generated reports conform to a standardized structure.

## Components

### 1. JSON Schema Definition

**File**: `data/testcase_results_container/schema.json`

- Comprehensive JSON Schema Draft 07 definition
- Validates the complete structure of container reports
- Enforces required fields, data types, and constraints
- Supports conditional validation (e.g., required fields for failed steps)

**Key Features**:
- Required fields at container, test result, sequence, and step levels
- Optional metadata fields (environment, platform, executor)
- String length constraints (title: 1-200 chars, project: 1-200 chars, etc.)
- Numeric minimum constraints (execution_duration ≥ 0, all counts ≥ 0)
- Enumerated step status values: `"pass"`, `"fail"`, `"not_executed"`
- ISO 8601 date-time format for test_date
- Additional properties disallowed at container level

### 2. Validation Logic in Verifier

**File**: `src/bin/verifier.rs`

**Function**: `validate_output_against_schema(output: &str, format: &str, schema_path: &PathBuf) -> Result<()>`

**CLI Parameter**:
- `--schema <PATH>`: Path to JSON schema file (default: `data/testcase_results_container/schema.json`)

**Process**:
1. Checks if schema file exists at the provided path (errors if not found)
2. Loads schema from the specified path
3. Compiles the JSON schema using the `jsonschema` crate
4. Parses the output (YAML converted to JSON if needed)
5. Validates the output against the compiled schema
6. Reports detailed validation errors with paths if validation fails

**Integration**:
- Automatic validation during report generation in `generate_output()` function
- Validation runs for both YAML and JSON output formats
- Schema path is configurable via `--schema` command-line option
- Errors if schema file is not found (required for validation)
- Provides detailed error messages on validation failure

**Example Output**:
```
INFO  verifier > ✓ Output validation passed: conforms to schema
INFO  verifier > Report written to report.yaml
```

**Error Output (Missing Schema)**:
```
Error: Schema file not found at data/testcase_results_container/schema.json. Please ensure the schema file exists or specify a valid path using --schema
```

### 3. Sample Data

**File**: `data/testcase_results_container/data_sample.yml`

- Comprehensive example of a valid container report
- Demonstrates all three step statuses (pass, fail, not_executed)
- Includes optional fields (requirement, item, tc)
- Shows metadata with all optional fields populated
- Can be used for testing and documentation

### 4. Documentation

**File**: `data/testcase_results_container/README.md`

Complete documentation including:
- Schema structure overview
- Validation process details
- Required and optional fields
- Data types and constraints
- Usage examples
- Troubleshooting guide
- Schema maintenance guidelines

### 5. Validation Script

**File**: `scripts/validate-container-output.sh`

Standalone validation script for manual testing:
- Validates YAML or JSON files against the schema
- Converts YAML to JSON for validation
- Uses Python's jsonschema library
- Provides detailed error reporting
- Integrates with the logging library

**Usage**:
```bash
./scripts/validate-container-output.sh data/testcase_results_container/data_sample.yml
```

### 6. Test Suite

**File**: `tests/container_schema_validation_test.rs`

Comprehensive test coverage:
- Schema is valid JSON Schema
- Sample data validates against schema
- Minimal valid container validates
- Missing required fields detected
- Invalid step status values detected
- Valid step statuses accepted (pass, fail, not_executed)
- String length constraints enforced
- Numeric minimum constraints enforced
- Optional fields can be omitted
- Additional properties not allowed

**Run Tests**:
```bash
cargo test container_schema_validation
```

## Schema Structure

### Container Level
```yaml
title: string (1-200 chars, required)
project: string (1-200 chars, required)
test_date: ISO 8601 datetime (required)
test_results: array (required)
metadata: object (required)
```

### Metadata
```yaml
environment: string (1-100 chars, optional)
platform: string (1-150 chars, optional)
executor: string (1-150 chars, optional)
execution_duration: number (≥0, required)
total_test_cases: integer (≥0, required)
passed_test_cases: integer (≥0, required)
failed_test_cases: integer (≥0, required)
```

### Test Result
```yaml
test_case_id: string (required)
description: string (required)
sequences: array (required)
total_steps: integer (≥0, required)
passed_steps: integer (≥0, required)
failed_steps: integer (≥0, required)
not_executed_steps: integer (≥0, required)
overall_pass: boolean (required)
requirement: string (optional)
item: integer (optional)
tc: integer (optional)
```

### Sequence
```yaml
sequence_id: integer (≥1, required)
name: string (required)
step_results: array (required)
all_steps_passed: boolean (required)
requirement: string (optional)
item: integer (optional)
tc: integer (optional)
```

### Step Result
```yaml
status: enum ["pass", "fail", "not_executed"] (required)
step: integer (≥1, required)
description: string (required)
requirement: string (optional)
item: integer (optional)
tc: integer (optional)

# When status = "fail", these are required:
expected:
  success: boolean or null
  result: string (required)
  output: string (required)
actual_result: string (required when fail)
actual_output: string (required when fail)
reason: string (required when fail)
```

## Usage Examples

### Generate and Validate Report

```bash
# YAML format with validation (uses default schema)
verifier -f logs/ --format yaml --output report.yaml

# JSON format with validation (uses default schema)
verifier -f logs/ --format json --output report.json

# With custom metadata
verifier -f logs/ --format yaml --output report.yaml \
  --title "Nightly Test Run" \
  --project "Production Suite" \
  --environment "Production" \
  --platform "Linux x86_64" \
  --executor "Jenkins v3.2"

# Using a custom schema file
verifier -f logs/ --format yaml --output report.yaml \
  --schema /path/to/custom/schema.json

# Using a custom schema with metadata
verifier -f logs/ --format json --output report.json \
  --schema /path/to/custom/schema.json \
  --title "Custom Test Run" \
  --environment "Staging"
```

### Manual Validation

```bash
# Validate a YAML file
./scripts/validate-container-output.sh data/testcase_results_container/data_sample.yml

# Validate a JSON file
./scripts/validate-container-output.sh report.json
```

## Error Handling

### Validation Errors

When validation fails, detailed error messages are provided:

```
ERROR verifier > ✗ Output validation failed:
ERROR verifier >   - 'title' is a required property at /
ERROR verifier >   - 'metadata' is a required property at /
Error: Output does not conform to schema:
  - 'title' is a required property at /
  - 'metadata' is a required property at /
```

### Missing Schema File

If the schema file is missing, the verifier will exit with an error:

```
Error: Schema file not found at data/testcase_results_container/schema.json. Please ensure the schema file exists or specify a valid path using --schema
```

To use a custom schema file:

```bash
verifier -f logs/ --format yaml --output report.yaml --schema /path/to/schema.json
```
