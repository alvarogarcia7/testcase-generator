# Test Acceptance Suite - Directory Structure

## Overview

The test acceptance suite uses a **numbered directory organization** to represent the sequential workflow of test validation, execution, verification, and documentation. This structure provides clear separation of concerns and makes the data flow through the testing pipeline immediately apparent.

The organization consists of **5 main directories**, numbered in increments of 5 to allow future expansion:

1. **00_test_cases** - Test case definitions (YAML files)
2. **05_scripts** - Generated test execution scripts
3. **10_test_results** - Execution logs and test outputs
4. **20_verification** - Verification results and containers
5. **30_documentation_source** - Generated documentation and reports

This structure supports both **automated testing workflows** and **manual test execution**, with validation scripts serving as orchestrators to coordinate the entire pipeline.

## Directory Reference

| Directory | Purpose | Contents | Gitignored |
|-----------|---------|----------|------------|
| **00_test_cases/** | Test case definitions | Test case YAML files organized by category (bash_commands, complex, dependencies, etc.) | No |
| **05_scripts/** | Test execution scripts | Auto-generated bash scripts from test case YAMLs | Yes |
| **10_test_results/** | Execution outputs | `execution_logs/` - JSON execution logs from test runs | Yes |
| **20_verification/** | Verification results | Verification container YAMLs (`*_container.yaml`) with test results and metadata | Yes |
| **30_documentation_source/** | Documentation | Generated reports in AsciiDoc and Markdown formats, summary files | Yes |

### Directory Details

#### 00_test_cases/

Contains all test case definitions in YAML format, organized by test category:

- `bash_commands/` - Bash command execution tests
- `complex/` - Complex multi-step test scenarios
- `dependencies/` - Dependency and prerequisite tests
- `hooks/` - Hook execution tests
- `hydration/` - Variable hydration tests
- `manual_steps/` - Manual step interaction tests
- `output_validation/` - Output validation tests
- `prerequisites/` - Prerequisites validation tests
- `tpdg_compatibility/` - TPDG format compatibility tests
- `variables/` - Variable passing and substitution tests

**Schema:** All YAML files validated against `schemas/test-case.schema.json`

#### 05_scripts/

Contains executable bash scripts generated from test case YAMLs. These scripts:

- Are auto-generated by `test-executor generate` command
- Use JSON logging format (`--json-log` flag)
- Output execution logs to `10_test_results/execution_logs/`
- Are recreated during each acceptance suite run

**Generation:** `validate_stage2_scripts.sh` orchestrator

#### 10_test_results/

Stores test execution outputs and logs:

- **execution_logs/** - JSON-formatted execution logs, one per test case
  - Named: `{test_case_basename}.json`
  - Schema: Structured JSON with steps, results, timestamps
  - Used by verifier for result validation

**Generation:** `validate_stage3_execution.sh` orchestrator

#### 20_verification/

Contains verification containers generated from execution logs:

- Container YAMLs combining test metadata, execution results, and environment info
- Named: `{test_case_basename}_container.yaml`
- Schema: `data/testcase_results_container/schema.json`
- Used as input for documentation generation

**Generation:** `validate_stage4_verification.sh` orchestrator

#### 30_documentation_source/

Stores generated documentation in multiple formats:

- **asciidoc/** - AsciiDoc format reports (`.adoc`)
- **markdown/** - Markdown format reports (`.md`)
- **acceptance_suite_summary.txt** - Summary report from acceptance suite run

**Generation:** `validate_stage5_tpdg_result_docs.sh`, `validate_stage6_tpdg_plan_docs.sh`, `validate_stage7_consolidated_docs.sh` orchestrators

## Usage Examples

### Running Individual Validation Scripts

Each stage has a dedicated orchestrator script that can be run independently:

#### Stage 1: Validate Test Case YAMLs

```bash
# Validate all test case YAML files
./test-acceptance/validate_stage1_yaml.sh

# Verbose mode
./test-acceptance/validate_stage1_yaml.sh --verbose

# Show help
./test-acceptance/validate_stage1_yaml.sh --help
```

**What it does:**
- Validates YAML files in `00_test_cases/` against schema
- Checks required fields, sequences, hooks, hydration vars, prerequisites
- Reports validation statistics and errors

#### Stage 2: Generate Test Scripts

```bash
# Generate test scripts from YAML files
./test-acceptance/validate_stage2_scripts.sh

# Verbose mode
./test-acceptance/validate_stage2_scripts.sh --verbose
```

**What it does:**
- Generates bash scripts in `05_scripts/` from `00_test_cases/` YAMLs
- Uses `test-executor generate --json-log`
- Makes scripts executable

#### Stage 3: Execute Test Scripts

```bash
# Execute all automated test scripts
./test-acceptance/validate_stage3_execution.sh

# Include manual tests (requires interaction)
./test-acceptance/validate_stage3_execution.sh --include-manual

# Verbose mode
./test-acceptance/validate_stage3_execution.sh --verbose
```

**What it does:**
- Executes scripts from `05_scripts/`
- Outputs JSON logs to `10_test_results/execution_logs/`
- Skips manual tests by default
- Validates JSON output structure

#### Stage 4: Verify Execution Results

```bash
# Generate verification containers
./test-acceptance/validate_stage4_verification.sh

# Verbose mode
./test-acceptance/validate_stage4_verification.sh --verbose
```

**What it does:**
- Processes execution logs from `10_test_results/execution_logs/`
- Generates container YAMLs in `20_verification/`
- Includes metadata (title, project, environment)
- Validates containers against schema

#### Stage 5-7: Generate Documentation

```bash
# Generate result documentation
./test-acceptance/validate_stage5_tpdg_result_docs.sh

# Generate plan documentation
./test-acceptance/validate_stage6_tpdg_plan_docs.sh

# Generate consolidated documentation
./test-acceptance/validate_stage7_consolidated_docs.sh

# Verbose mode
./test-acceptance/validate_stage5_tpdg_result_docs.sh --verbose
```

**What it does:**
- Processes containers from `20_verification/`
- Generates AsciiDoc and Markdown reports in `30_documentation_source/`
- Uses test-plan-documentation-generator (TPDG)
- Creates summary reports

### Running the Complete Acceptance Suite

The master orchestrator runs all stages sequentially:

```bash
# Run complete acceptance suite
./test-acceptance/run_acceptance_suite.sh

# Verbose output
./test-acceptance/run_acceptance_suite.sh --verbose

# Include manual tests
./test-acceptance/run_acceptance_suite.sh --include-manual

# Skip specific stages (for debugging)
./test-acceptance/run_acceptance_suite.sh --skip-generation
./test-acceptance/run_acceptance_suite.sh --skip-execution
./test-acceptance/run_acceptance_suite.sh --skip-verification
./test-acceptance/run_acceptance_suite.sh --skip-documentation

# Combined options
./test-acceptance/run_acceptance_suite.sh --verbose --skip-documentation

# Show help
./test-acceptance/run_acceptance_suite.sh --help
```

**Output:**
- Runs all 6 stages sequentially
- Generates comprehensive summary report
- Reports saved to `30_documentation_source/acceptance_suite_summary.txt`
- Exit code 0 = success, 1 = failures detected

### Quick Validation Workflow

```bash
# 1. Validate YAMLs
./test-acceptance/validate_stage1_yaml.sh

# 2. Generate and execute tests
./test-acceptance/validate_stage2_scripts.sh
./test-acceptance/validate_stage3_execution.sh

# 3. Verify results
./test-acceptance/validate_stage4_verification.sh

# 4. Generate documentation
./test-acceptance/validate_stage5_tpdg_result_docs.sh
```

Or run everything at once:

```bash
# Full acceptance suite
./test-acceptance/run_acceptance_suite.sh --verbose
```

## Migration Notes

### Backward Compatibility via Symlinks

The numbered directory structure was introduced to replace the legacy flat structure. To maintain backward compatibility, **symlinks** are created from old directory names to new locations:

| Legacy Name | Symlink Target | Purpose |
|-------------|----------------|---------|
| `test_cases` → | `00_test_cases` | Test case YAML files |
| `scripts` → | `05_scripts` | Generated test scripts |
| `execution_logs` → | `10_test_results/execution_logs` | Execution log files |

### Migration Process

The migration is handled by the `migrate_to_organized_structure.sh` script:

```bash
# Preview migration changes (dry run)
./test-acceptance/migrate_to_organized_structure.sh --dry-run

# Execute migration
./test-acceptance/migrate_to_organized_structure.sh

# Show help
./test-acceptance/migrate_to_organized_structure.sh --help
```

**Migration steps:**

1. **Create numbered directories** (`00_test_cases`, `05_scripts`, `10_test_results`, `20_verification`, `30_documentation_source`)
2. **Move existing content:**
   - `test_cases/` → `00_test_cases/`
   - `scripts_test/` → `05_scripts/`
   - `results/logs/` → `10_test_results/execution_logs/`
   - `verification_results/` → `20_verification/` (if exists)
   - `reports/` → `30_documentation_source/` (if exists)
3. **Create symlinks:**
   - `test_cases` → `00_test_cases`
   - `scripts` → `05_scripts`
   - `execution_logs` → `10_test_results/execution_logs`
4. **Validate structure** (all directories and symlinks in place)

### Why Symlinks?

Symlinks provide **zero-downtime migration**:

- **Existing scripts continue to work** (reference old paths)
- **New scripts use numbered paths** (clear organization)
- **Gradual transition** (update references incrementally)
- **No code changes required** (backward compatible)

### Verifying Migration

After migration, validate the structure:

```bash
# Check directory structure
ls -la test-acceptance/

# Verify symlinks
ls -la test-acceptance/ | grep '^l'

# Expected symlinks:
# test_cases -> 00_test_cases
# scripts -> 05_scripts
# execution_logs -> 10_test_results/execution_logs
```

### Updating References

While symlinks maintain compatibility, **it's recommended to update references** to use numbered directories:

**Before:**
```bash
TEST_CASES_DIR="$SCRIPT_DIR/test_cases"
SCRIPTS_DIR="$SCRIPT_DIR/scripts"
LOGS_DIR="$SCRIPT_DIR/execution_logs"
```

**After:**
```bash
TEST_CASES_DIR="$SCRIPT_DIR/00_test_cases"
SCRIPTS_DIR="$SCRIPT_DIR/05_scripts"
LOGS_DIR="$SCRIPT_DIR/10_test_results/execution_logs"
```

### Future Expansion

The numbered structure (increments of 5) allows for future stages:

- `02_preprocessed_cases` - Preprocessed test cases
- `07_prepared_environments` - Test environment setup
- `15_intermediate_results` - Intermediate processing results
- `25_audit_logs` - Audit trail and compliance logs
- `35_archived_results` - Historical test results

This flexibility supports evolving testing workflows without reorganization.

## Orchestrator Scripts

The validation scripts (`validate_stage*.sh`) serve as **orchestrators** that coordinate the testing pipeline. They are responsible for:

### Orchestrator Responsibilities

1. **Stage Coordination**
   - Execute binaries in correct order
   - Pass data between stages
   - Validate inputs/outputs

2. **Binary Management**
   - Locate required binaries (`validate-yaml`, `test-executor`, `verifier`, `test-plan-documentation-generator`)
   - Verify binary availability
   - Provide clear error messages if missing

3. **Directory Management**
   - Create output directories
   - Organize generated files
   - Clean up temporary files

4. **Progress Reporting**
   - Track pass/fail counts
   - Display stage progress
   - Generate summary reports

5. **Error Handling**
   - Continue on failure (best effort)
   - Track all errors
   - Report comprehensive results

6. **Flexibility**
   - Support verbose mode
   - Allow manual test inclusion
   - Enable stage skipping

### Master Orchestrator

The `run_acceptance_suite.sh` script is the **master orchestrator** that:

- Runs all validation stages (1-7) sequentially
- Tracks statistics across stages
- Generates comprehensive summary report
- Supports skip flags for debugging
- Provides single entry point for full validation

**Usage:**
```bash
./test-acceptance/run_acceptance_suite.sh [OPTIONS]

OPTIONS:
    --verbose            Enable verbose output
    --include-manual     Include manual tests (requires interaction)
    --skip-generation    Skip stage 2 (script generation)
    --skip-execution     Skip stage 3 (test execution)
    --skip-verification  Skip stages 4-5 (verification + validation)
    --skip-documentation Skip stage 6 (documentation generation)
    -h, --help          Show help message
```

✓ **Clear organization** - Each directory represents a stage in the workflow  
✓ **Intuitive flow** - Numbers indicate processing order  
✓ **Future-proof** - Room for new stages (5-increment numbering)  
✓ **Backward compatible** - Symlinks preserve existing references  
✓ **Orchestrated** - Validation scripts coordinate the entire pipeline  
✓ **Comprehensive** - All artifacts organized by stage  

This structure supports both automated CI/CD workflows and manual testing scenarios, with clear separation between source definitions, generated artifacts, execution results, verification outputs, and documentation.
