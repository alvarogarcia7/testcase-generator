# Verification Results (Stage 4 Output)

This directory contains verification YAML files that demonstrate the structure and format of test case verification results.

## Directory Purpose

The `20_verification/` directory is the output directory for Stage 4 (Verification) of the acceptance test suite pipeline. Files in this directory:

- Contain verification results from running the `verifier` tool on execution logs
- Follow the `test_verification` schema defined in `tcms/test-verification.schema.v1.json`
- Document pass/fail status of test cases with detailed step-level results
- Include metadata and statistics about test execution

## Schema Reference

All verification YAML files must follow the schema: `tcms/test-verification.schema.v1.json`

### Required Fields

Every verification YAML file must include:

- `type`: Fixed value `test_verification`
- `schema`: Fixed value `tcms/test-verification.schema.v1.json`
- `test_case_id`: Identifier of the test case being verified
- `description`: Description of the test case
- `sequences`: Array of sequence verification results
- `total_steps`: Total number of steps across all sequences
- `passed_steps`: Number of steps that passed
- `failed_steps`: Number of steps that failed
- `not_executed_steps`: Number of steps not executed
- `overall_pass`: Boolean indicating overall pass/fail status

### Optional Fields

- `requirement`: Requirement identifier for reporting
- `item`: Item number for reporting
- `tc`: TC number for reporting

## File Naming Convention

Verification files follow the naming pattern:
```
<TEST_CASE_ID>_verification.yaml
```

Examples:
- `TC_SUCCESS_SIMPLE_001_verification.yaml`
- `TC_FAILURE_FIRST_STEP_001_verification.yaml`
- `TC_SUCCESS_MULTI_SEQ_001_verification.yaml`

## Step Result Types

Each step in a sequence can have one of three verification results:

### Pass - Step Passed Verification

```yaml
- Pass:
    step: 1
    description: Step description
    requirement: REQ_ID          # optional
    item: 1                      # optional
    tc: 1                        # optional
```

### Fail - Step Failed Verification

```yaml
- Fail:
    step: 1
    description: Step description
    expected:
      success: true
      result: '0'
      output: expected output
    actual_result: '1'           # actual exit code/result
    actual_output: actual        # actual command output
    reason: detailed failure reason
    requirement: REQ_ID          # optional
    item: 1                      # optional
    tc: 1                        # optional
```

### NotExecuted - Step Not Executed

```yaml
- NotExecuted:
    step: 2
    description: This step did not execute
```

## Example Files

### 1. TC_SUCCESS_SIMPLE_001_verification.yaml

Demonstrates a successful test case with a single sequence and 3 passing steps:
- All steps passed
- overall_pass: true
- Used as baseline for successful test verification

### 2. TC_FAILURE_FIRST_STEP_001_verification.yaml

Demonstrates a test case where the first step fails:
- Step 1 fails verification
- Steps 2 and 3 are not executed (cascade failure)
- overall_pass: false
- Shows failure propagation across sequences

### 3. TC_SUCCESS_MULTI_SEQ_001_verification.yaml

Demonstrates a more complex test case with 3 sequences:
- Sequence 1: File Creation (3 steps, all pass)
- Sequence 2: File Content Processing (4 steps, all pass)
- Sequence 3: File Cleanup (3 steps, all pass)
- total_steps: 10, all passed
- overall_pass: true

## Usage in Pipeline

These verification YAML files are:

1. **Generated by**: Stage 4 verification process (using the `verifier` binary)
2. **Validated by**: Stage 5 container validation (against JSON schemas)
3. **Consumed by**: Stage 6-7 documentation generation (to create reports)

## Creating New Verification Files

When creating new verification YAML files:

1. Start with a test case file (e.g., `testcases/example.yaml`)
2. Execute the test case to generate an execution log
3. Run the verifier on the execution log to generate the container YAML
4. Extract the test case verification result from the container
5. Save as a standalone verification YAML file in this directory

## Schema Validation

Verify that your verification YAML files comply with the schema:

```bash
# Using cargo
cargo run -p validate-yaml -- 20_verification/TC_SUCCESS_SIMPLE_001_verification.yaml

# Using the validate-yaml binary
validate-yaml 20_verification/TC_SUCCESS_SIMPLE_001_verification.yaml
```

## Statistics Fields

The verification file tracks the following statistics:

- **total_steps**: Sum of all steps across all sequences
- **passed_steps**: Number of steps with Pass result
- **failed_steps**: Number of steps with Fail result
- **not_executed_steps**: Number of steps with NotExecuted result

These should sum to total_steps:
```
passed_steps + failed_steps + not_executed_steps = total_steps
```
